# i18n

## Mission
Implement comprehensive internationalization (i18n) and localization (l10n) support. Handle translation setup, string extraction, ICU MessageFormat, locale management, date/number formatting, RTL layouts, and server-side i18n for SEO.

## First Action
Read `docs/ARCHITECTURE.md` if it exists, then scan the project for existing i18n setup (config files, locale directories, translation keys, `t()` or `$t()` calls, RTL stylesheets).

## Core Principles

### Security First (Mandatory)
- NEVER trust user input - validate and sanitize ALL inputs on server side
- ALWAYS use parameterized queries - never string concatenation for SQL/NoSQL
- NEVER expose sensitive data (tokens, passwords, PII) in logs, URLs, or error messages
- ALWAYS implement rate limiting on public endpoints
- Use HTTPS everywhere, set secure headers (CSP, HSTS, X-Frame-Options)
- Follow OWASP Top 10 - prevent XSS, CSRF, injection, broken auth, etc.
- Secrets in environment variables only - never hardcode

### Performance First (Mandatory)
- ALWAYS use TanStack Query (React Query / Vue Query) for server state caching
- Set appropriate `staleTime` and `gcTime` for each query based on data freshness needs
- Use `keepPreviousData` for pagination to avoid loading flickers
- Implement optimistic updates for mutations when UX benefits
- Use proper cache invalidation (`invalidateQueries`) - stale UI is a bug
- Lazy load routes, components, and heavy dependencies
- Avoid N+1 queries - batch requests, use proper data loading patterns

### Code Language (Mandatory)
- ALWAYS write code (variables, functions, comments, commits) in English
- Only use other languages if explicitly requested by the user
- User-facing text (UI labels, messages) should match project's i18n strategy

## Scope Detection
- **Setup**: user wants i18n library configuration, locale files, framework integration → Setup mode
- **Extract**: user wants to extract hardcoded strings, create translation keys, automate extraction → Extract mode
- **Quality**: user wants missing translation detection, coverage reports, pseudo-localization, i18n testing → Quality mode
- **RTL**: user wants right-to-left layout support, bidirectional text, logical CSS properties → RTL mode

### Framework Integration

**React (react-i18next)**
```typescript
// i18n/config.ts
import i18n from 'i18next'
import { initReactI18next } from 'react-i18next'
import Backend from 'i18next-http-backend'
import LanguageDetector from 'i18next-browser-languagedetector'

i18n
  .use(Backend)
  .use(LanguageDetector)
  .use(initReactI18next)
  .init({
    fallbackLng: 'en',
    ns: ['common', 'auth', 'dashboard'],
    defaultNS: 'common',
    interpolation: { escapeValue: false },
    backend: {
      loadPath: '/locales/{{lng}}/{{ns}}.json',
    },
  })
```

**Vue (vue-i18n)**
```typescript
// i18n/config.ts
import { createI18n } from 'vue-i18n'

export const i18n = createI18n({
  legacy: false,
  locale: 'en',
  fallbackLocale: 'en',
  messages: {
    en: { common: await import('./locales/en/common.json') },
  },
})
```

### ICU MessageFormat

Use ICU MessageFormat for complex pluralization and gender selection instead of simple key variants:

```
{count, plural,
  =0 {No items}
  one {# item}
  other {# items}}
```

```
{gender, select,
  male {He}
  female {She}
  other {They}} liked your post
```

```
{count, plural,
  =0 {No messages}
  one {You have # new message}
  other {You have # new messages}} from {name}
```

Libraries: `intl-messageformat` (standalone), `i18next-icu` (i18next plugin), `vue-i18n` (built-in ICU support with `messageCompiler`).

### Lazy Loading Translations Per Route

```typescript
// React: lazy load namespace on route
const Dashboard = lazy(() =>
  i18n.loadNamespaces('dashboard').then(() => import('./pages/Dashboard'))
)

// Vue Router: load locale in beforeEnter
{
  path: '/dashboard',
  component: () => import('./pages/Dashboard.vue'),
  beforeEnter: async () => {
    await loadLocaleMessages(i18n, 'dashboard')
  },
}
```

### Namespace Strategy for Large Apps

```text
locales/
├── en/
│   ├── common.json       # Shared: buttons, labels, errors
│   ├── auth.json          # Login, register, password reset
│   ├── dashboard.json     # Dashboard-specific strings
│   ├── settings.json      # Settings page
│   └── emails.json        # Email templates
├── pt/
│   └── ...
```

Rule: One namespace per feature area. `common` namespace loads globally, others load per route.

### Rules
- ALWAYS configure a fallback locale chain

- Use ICU MessageFormat for plurals and gender - never concatenate strings
- Store translation management tokens in environment variables only

## Extract Mode

### Workflow
1. Scan codebase for hardcoded user-facing strings (JSX text, template literals, attribute values)
2. Generate translation keys following naming convention
3. Replace hardcoded strings with `t()` / `$t()` calls
4. Create/update locale JSON files with extracted keys
5. Add context comments for translators
6. Validate: grep for remaining hardcoded strings

### Key Naming Convention

```text
Pattern: [namespace].[context].[key]

Examples:
  common.actions.save          → "Save"
  common.actions.cancel        → "Cancel"
  auth.form.emailLabel         → "Email address"
  auth.form.emailPlaceholder   → "Enter your email"
  auth.errors.invalidPassword  → "Invalid password"
  dashboard.stats.totalUsers   → "Total users"
  errors.validation.required   → "This field is required"
```

Rules: lowercase keys, dot-separated hierarchy, descriptive context in the middle segment.

### Automated Extraction Tooling

| Tool | Framework | Command |
|------|-----------|---------|
| `i18next-scanner` | React/i18next | `i18next-scanner --config i18next-scanner.config.js` |
| `@formatjs/cli` | React/react-intl | `formatjs extract 'src/**/*.tsx' --out-file lang/en.json` |
| `vue-i18n-extract` | Vue | `vue-i18n-extract report --vueFiles './src/**/*.vue' --languageFiles './locales/*.json'` |

### Context Comments for Translators

```json
{
  "auth.form.emailLabel": "Email address",
  "auth.form.emailLabel_context": "Label for the email input field on login/register forms",
  "common.actions.close": "Close",
  "common.actions.close_context": "Button to close a modal or dialog, keep short (max 10 chars)"
}
```

For react-intl, use `description` in `defineMessages`:
```typescript
defineMessages({
  greeting: {
    id: 'home.greeting',
    defaultMessage: 'Welcome back, {name}!',
    description: 'Greeting shown on homepage after login. {name} is user first name.',
  },
})
```

### Handling Dynamic Strings and Interpolation

```typescript
// WRONG: concatenation breaks word order in other languages
t('greeting') + ', ' + name + '!'

// RIGHT: interpolation
t('greeting', { name })
// Key: "Welcome back, {{name}}!"

// WRONG: splitting sentences across keys
t('youHave') + count + t('items')

// RIGHT: ICU plural in a single key
t('itemCount', { count })
// Key: "{count, plural, =0 {No items} one {# item} other {# items}}"
```

### Rules
- NEVER split sentences across multiple translation keys
- ALWAYS use interpolation for dynamic values - never concatenate
- ALWAYS add context comments for ambiguous strings
- Key names in English, values in each target language
- Run extraction as part of CI to catch new hardcoded strings

## Quality Mode

### Workflow
1. Run missing translation detection across all locales
2. Generate translation key coverage report
3. Run pseudo-localization to test layout flexibility
4. Validate ICU syntax in all translation files
5. Set up i18n-specific tests
6. Detect and remove unused translation keys

### Missing Translation Detection

```typescript
// Script: check-translations.ts
import en from './locales/en/common.json'
import pt from './locales/pt/common.json'

const enKeys = Object.keys(flattenObject(en))
const ptKeys = Object.keys(flattenObject(pt))
const missing = enKeys.filter(k => !ptKeys.includes(k))
const unused = ptKeys.filter(k => !enKeys.includes(k))

if (missing.length) {
  console.error(`Missing in pt: ${missing.join(', ')}`)
  process.exit(1)
}
```

Run in CI to block merges with incomplete translations.

### Pseudo-Localization

Replace characters with accented equivalents and add padding to detect:
- Strings that overflow their containers
- Hardcoded strings that bypassed extraction
- Layout issues with longer text

```text
Original: "Save changes"
Pseudo:   "[Ŝȧṽḗḗ ƈħȧȧƞɠḗḗş________]"
```

Tools: `pseudo-localization` npm package, `i18next` pseudo plugin, custom script.

### ICU Syntax Validation

```bash
# Validate all ICU messages parse correctly
npx @formatjs/cli compile 'locales/en/**/*.json' --ast
```

Catch syntax errors (unclosed braces, invalid plural categories) before they reach production.

### Testing i18n

| Test Type | What to Test | Tool |
|-----------|-------------|------|
| **Unit** | Date/number/currency formatters produce correct output per locale | Vitest/Jest |
| **Unit** | Plural rules return correct form for 0, 1, 2, 5, 21 | Vitest/Jest |
| **Integration** | Rendered components show translated text after locale switch | Testing Library |
| **Visual regression** | RTL layouts, long text overflow, character rendering | Playwright/Chromatic |
| **E2E** | Full locale switching flow, URL-based locale, persistence | Playwright/Cypress |

### Detecting Unused Translation Keys

```bash
# Scan source for key usage, compare with locale files
npx i18next-scanner --config scanner.config.js
# Or custom: grep all t() calls, diff against locale keys
```

Remove unused keys regularly to keep translation files maintainable and reduce translator workload.

### Rules
- Missing translation check MUST run in CI - broken translations are user-facing bugs
- Pseudo-localization MUST be tested before adding a new locale
- ICU syntax validation MUST pass before deploy
- Test formatters with edge-case locales (ar, ja, de) not just en/es

## RTL Mode

### Workflow
1. Audit current CSS for physical direction properties (margin-left, padding-right, text-align: left, float: left)
2. Replace with logical CSS properties
3. Configure bidirectional text handling
4. Set up icon/image mirroring rules
5. Test with RTL locale (ar, he, fa)
6. Validate: visual regression tests for RTL layouts

### Logical CSS Properties Reference

| Physical (LTR only) | Logical (LTR + RTL) |
|---------------------|---------------------|
| `margin-left` | `margin-inline-start` |
| `margin-right` | `margin-inline-end` |
| `padding-left` | `padding-inline-start` |
| `padding-right` | `padding-inline-end` |
| `text-align: left` | `text-align: start` |
| `text-align: right` | `text-align: end` |
| `float: left` | `float: inline-start` |
| `left: 0` | `inset-inline-start: 0` |
| `right: 0` | `inset-inline-end: 0` |
| `border-left` | `border-inline-start` |
| `border-right` | `border-inline-end` |
| `width` | `inline-size` |
| `height` | `block-size` |

### Base RTL Setup

```css
/* Set direction on html element */
html[dir="rtl"] {
  direction: rtl;
}

/* Use logical properties everywhere */
.card {
  margin-inline-start: 1rem;
  padding-inline-end: 0.5rem;
  border-inline-start: 3px solid var(--accent);
  text-align: start;
}
```

### Bidirectional Text Handling

```html
<!-- Auto-detect direction for user-generated content -->
<p dir="auto">{{ userInput }}</p>

<!-- Isolate embedded opposite-direction text -->
<p>Order #123 for <bdi>{{ arabicUserName }}</bdi> confirmed</p>
```

CSS `unicode-bidi` values:
- `normal` - default behavior
- `embed` - open a new level of embedding
- `isolate` - isolate from surrounding text (preferred for inline mixed-direction text)
- `bidi-override` - force direction (use sparingly)

### Icon and Image Mirroring

| Mirror (directional) | Do NOT mirror (universal) |
|----------------------|--------------------------|
| Arrows (back, forward, next) | Search icon |
| Navigation chevrons | Close (X) icon |
| Text alignment icons | Checkmarks |
| Progress indicators | Media controls (play/pause) |
| Chat bubble tails | Logos and branding |
| Undo/redo icons | Clock icons |

```css
/* Mirror directional icons in RTL */
[dir="rtl"] .icon-arrow,
[dir="rtl"] .icon-chevron,
[dir="rtl"] .icon-back {
  transform: scaleX(-1);
}
```

### Testing RTL with Playwright

```typescript
// playwright.config.ts - add RTL project
{
  name: 'rtl',
  use: {
    locale: 'ar',
    // Set direction via page script
  },
}

// test
test('layout renders correctly in RTL', async ({ page }) => {
  await page.evaluate(() => document.documentElement.setAttribute('dir', 'rtl'))
  await page.goto('/dashboard')
  await expect(page.locator('.sidebar')).toHaveCSS('inset-inline-start', '0px')
  await expect(page).toHaveScreenshot('dashboard-rtl.png')
})
```

### Rules
- ALWAYS use logical CSS properties - never physical left/right for layout
- ALWAYS set `dir` attribute on `<html>` element based on locale
- Use `dir="auto"` for user-generated content
- Mirror only directional icons, never universal icons or logos
- Test RTL with real content, not just mirrored LTR - Arabic text has different line heights and character widths

## Server-Side i18n

### SSR/SSG Configuration

```typescript
// Detect locale from request
function detectLocale(req: Request): string {
  // 1. URL segment: /pt/about → "pt"
  const urlLocale = req.url.split('/')[1]
  if (SUPPORTED_LOCALES.includes(urlLocale)) return urlLocale

  // 2. Cookie
  const cookieLocale = parseCookies(req).locale
  if (SUPPORTED_LOCALES.includes(cookieLocale)) return cookieLocale

  // 3. Accept-Language header
  const headerLocale = parseAcceptLanguage(req.headers['accept-language'])
  if (SUPPORTED_LOCALES.includes(headerLocale)) return headerLocale

  return DEFAULT_LOCALE
}
```

**Locale detection order:** URL segment → cookie → `Accept-Language` header → default locale.

### SEO: hreflang and Locale URLs

```html
<!-- In <head> for every page -->
<link rel="alternate" hreflang="en" href="https://example.com/en/about" />
<link rel="alternate" hreflang="pt" href="https://example.com/pt/about" />
<link rel="alternate" hreflang="ar" href="https://example.com/ar/about" />
<link rel="alternate" hreflang="x-default" href="https://example.com/about" />
```

**URL strategies:** `/en/page` (subdirectory, recommended), `en.example.com` (subdomain), `example.com.br` (ccTLD).

### Static Generation Per Locale

```typescript
// Next.js: generate static pages for each locale
export async function getStaticPaths() {
  const locales = ['en', 'pt', 'es', 'ar']
  const paths = locales.map(locale => ({ params: { locale } }))
  return { paths, fallback: false }
}

// Astro: generate locale routes
export function getStaticPaths() {
  return LOCALES.map(locale => ({
    params: { lang: locale },
    props: { translations: loadTranslations(locale) },
  }))
}
```

Generate a sitemap entry per locale with `hreflang` references for search engines.

### Rules
- URL-based locale is preferred for SEO - search engines can index each locale separately
- ALWAYS include `hreflang` tags on all pages with `x-default` fallback
- Locale detection MUST follow the order: URL → cookie → header → default
- Generate sitemap with all locale variants

## Verification Protocol

| Claim | Required Proof |
|-------|---------------|
| "i18n is configured" | Locale switching works, fallback locale loads, no console errors |
| "Strings extracted" | Grep for hardcoded strings returns zero user-facing hits |
| "Translations complete" | Missing translation check script passes for all target locales |
| "ICU messages work" | Plural forms tested with 0, 1, 2, 5, 21; gender select renders correctly |
| "RTL works" | Screenshot comparison of LTR and RTL layouts, no overlapping or misaligned elements |
| "Server-side i18n works" | Page source contains correct `hreflang` tags, locale detected from URL/header |
| "Lazy loading works" | Network tab shows translation files loaded only when route is visited |

**Iron Law:** NEVER claim i18n is complete without testing with at least one non-Latin locale (ar, ja, zh). English-only testing is not i18n testing.

## Anti-Rationalization Table

| Excuse | Reality |
|--------|---------|
| "We only need English for now" | i18n is 10x harder to retrofit. Set up the structure now, add languages later. |
| "Just use string concatenation" | Concatenation breaks in every language with different word order. Use ICU MessageFormat. |
| "RTL can wait" | RTL affects every component. Logical CSS properties cost nothing upfront and save rewriting everything later. |
| "Machine translation is good enough" | For production? Never. Machine translation creates confused users and brand damage. |
| "We'll extract strings later" | Every hardcoded string is a debt. Extract while writing, not after 10,000 strings exist. |

## General Rules
- Framework-agnostic - works with any stack
- Reads ARCHITECTURE.md if present and follows existing conventions
- All user-facing strings must go through the translation function - no exceptions
- Always configure a fallback locale chain
- Use Intl API (`Intl.DateTimeFormat`, `Intl.NumberFormat`, `Intl.RelativeTimeFormat`) for formatting - never custom date/number formatters
- Provide `.env.example` for any translation management API keys
